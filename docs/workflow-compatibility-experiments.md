# Workflow Compatibility Experiments

This matrix compares the currently installed official Claude Code binary dynamic workflow behavior with this repository's workflow implementation. The baseline was originally observed on 2.1.150 and was refreshed after updating `/opt/homebrew/bin/claude` to 2.1.165.

## Official baseline observed

Sources: `https://code.claude.com/docs/en/workflows` and `https://claude.com/blog/introducing-dynamic-workflows-in-claude-code`.

- `/opt/homebrew/bin/claude --version`: `2.1.165 (Claude Code)` after the user forced an official binary update.
- Earlier baseline notes in this document were collected against `2.1.150 (Claude Code)`; current compatibility runs should treat `2.1.165` as the primary binary baseline.
- Official docs say dynamic workflows require Claude Code v2.1.154 or later and are research preview.
- The installed 2.1.150 binary contains hidden workflow strings and bundled workflow definitions, but `claude -p --bare` did not expose the hidden `Workflow` tool in the init tool list.
- Secondary binary probes found embedded built-in workflow names: `bugfix`, `bughunt`, `bughunt-lite`, `dashboard`, `deep-research`, `docs`, `investigate`, `plan-hunter`, and `review-branch`; `autopilot` appears in descriptive binary strings as an end-to-end task runner.
- Feature probes with `CLAUDE_CODE_WORKFLOWS`, `tengu_workflows_enabled`, and `CLAUDE_CODE_RECOVER_FEATURES=WORKFLOW_SCRIPTS` exposed some workflow-related slash command names only in this repository context, but did not expose the hidden `Workflow` tool for direct non-interactive use.
- Binary strings show official `Workflow` accepts a workflow name or `{ script, name, scriptPath }`, persists the script under the session directory, and returns `scriptPath` and `workflowRunId` for edit-and-resume flows.
- Binary strings and docs show official workflow runtime state includes `workflow_progress`, `workflow_agent`, `workflow_phase`, `workflow_log`, and `task_local_workflow*` events.
- The 2.1.165 binary exposes a larger dynamic workflow surface in strings: `WorkflowTool`, `WorkflowDetailDialog`, `WorkflowPermissionDialog`, `workflowNeedsUsageConsentPrompt`, `skipWorkflowAgent`, `retryWorkflowAgent`, `pauseWorkflowTask`, `killWorkflowTask`, `remote-workflow`, `workflow_remote_agent`, `workflow-abort`, `tengu_workflow_agent_cap_exceeded`, `tengu_workflow_budget_cap_exceeded`, and explicit `CLAUDE_CODE_DISABLE_WORKFLOWS` / `enableWorkflows` / `disableWorkflows` settings.
- The 2.1.165 help output does not list a top-level `workflows` command, but binary strings still include `/workflows` and dynamic workflow detail/progress UI components, indicating the new UI is likely behind settings, plan availability, or interactive runtime state.
- The refreshed metadata export against 2.1.165 currently observes bundled workflow names `autopilot`, `bugfix`, `bughunt`, `dashboard`, `deep-research`, `docs`, and `investigate`; older 2.1.150 probes also saw `bughunt-lite`, `plan-hunter`, and `review-branch`, so name coverage should be version-scoped in reports.
- Official runtime disallows `Date.now()`, `new Date()`, and `Math.random()` in scripts because they break resume.
- Refreshed deobfuscation now preserves long bundled workflow candidates instead of truncating at 2.4 KB; the recovered `deep-research` candidate includes `REPORT_SCHEMA`, official prompt fragments, URL normalization via `new URL()`, search-result relevance ranking, top-15 fetch budgeting, source-quality taxonomy (`primary`, `secondary`, `blog`, `forum`, `unreliable`), top-25 claim verification, quorum handling for skipped votes, synthesis fallback, and stats fields.
- Official-style workflow scripts may use deterministic web platform globals such as `URL`; local workflow script sandbox now exposes `URL` while continuing to omit Node globals like `process` and `require`.
- The official blog emphasizes long-running workflows that can run for hours or days, save progress, continue after interruptions, and iterate until build/test/review results converge.
- The blog examples emphasize independent solution attempts plus reviewers/refuters, including two-reviewer-per-file migration patterns and build/test repair loops.

## Interactive tmux evidence refreshed on 2026-06-06

- Empty `/workflows` panel parity: `scripts/workflow-ui-byte-compare.mjs` captured official and local panels under `.claude/workflow-ui-byte-compare/`; `workflowPanelByteEqual=true` for `Dynamic workflows / No dynamic workflows in this session / Esc to close`.
- Slash argument parity: tmux captures under `.claude/workflow-tmux-compare4/` and `.claude/workflow-tmux-ultracode-compare/` show official `/workflows`, `/workflows list`, `/workflows show compatibility-smoke`, and `/workflows dry-run compatibility-smoke` all open the same dynamic workflow panel. Local interactive `/workflows` now follows that visible behavior instead of routing arguments to text subcommands.
- `ultracode` prompt indicator parity: `.claude/workflow-tmux-ultracode-compare3/` shows official and local both render `Dynamic workflow requested for this turn · opt+w to ignore` when `ultracode ...` is typed at the prompt, after suppressing unrelated startup notifications.
- `ultracode` decision-path evidence: official strings state `Enable the "ultracode" keyword trigger: including the keyword in a prompt opts that turn into the Workflow tool. Set to false to disable the trigger. Default: true.` Local behavior maps the keyword to a per-turn `ultracode` effort and injects an explicit system prompt telling the model to prefer the `Workflow` tool for complex multi-agent work while preserving normal permissions; this is evidence-aligned but not a byte-for-byte reconstruction of closed-source model heuristics.
- Workflow usage permission prompt parity: `.claude/workflow-tmux-permission-compare6/` shows local `Workflow({ name: 'compatibility-smoke', args: 'permission smoke' })` renders the official-style title, saved workflow description, phase preview, script excerpt, args, consent options, `/workflows` usage-control text, `Esc to cancel · Tab to amend`, and `ctrl+g to edit script in $EDITOR`.
- Running workflow UI probe: `.claude/workflow-running-local-probe/before-accept.ansi` records a local tmux attempt to launch `Workflow({ name: 'compatibility-smoke', args: 'running smoke' })` from a loose cwd. The probe first exposed a local `Workflow` input schema regression (`type: "None"` when trying a string/object union), which was reverted because Anthropic tool schemas must remain object-shaped. A deterministic local running-detail tmux baseline is now captured by `scripts/workflow-running-detail-capture.mjs` at `.claude/workflow-running-detail-capture/local-running-detail.txt`, covering run ID, script path, runtime, progress, phase rows, recent events, and cancel/retry/skip controls. Full official running-detail bytes remain unverified because live official workflow launching still depends on model/tool cooperation.
- Official running-state string evidence from 2.1.165 includes `WorkflowDetailDialog`, `workflow_progress`, `workflow_agent`, `workflow_phase`, `workflow_progress_preview`, `Dynamic workflow cancelled`, `tengu_workflow_phase_completed`, `Dynamic workflows`, `No dynamic workflows in this session.`, and `Esc to close`.
- Official live running workflow detail bytes are now captured under `.claude/workflow-official-running-probe/`: `workflows-list.txt` shows the phase-selected detail view (`running-probe`, description, `0/1 agent`, `╭ Phases ────────┬ Wait · 1 agent`, `❯ 1 Wait 0/1`, `⏺ wait`, `gpt-5.5[1m]`, `0 tok · 1 tool`, controls `↑↓ select · x stop workflow · p pause · esc back · s save`), and `workflows-detail.txt` shows the agent-selected view with controls `↑↓ select · x stop · r restart · p pause · esc back · s save`. Local `workflowDetailSnapshot` now uses the same two-column phase/agent structure and control wording for running workflows; exact byte parity is still not claimed because local deterministic snapshots include local-only recent event rows and fixture model names.
- Scheduler/control string evidence in `.claude/workflow-deobfuscation/second-pass-large/WorkflowDetailDialog.window.js` and `workflow-symbol-neighborhoods.txt` shows official `registerWorkflowTask`, `updateWorkflowProgressBatch`, `pauseWorkflowTask`, `killWorkflowTask`, `failWorkflowTask`, `completeWorkflowTask`, `buildResumePrompt`, `agentControllers`, `progressVersion`, `defaultModel`, `task_local_workflow_resume`, `workflow-abort`, `lastAttemptReason`, `lastToolSummary`, `queuedAt`, and `lastProgressAt`. Local task/session state now records `progressVersion`, `defaultModel`, pause resume prompts, pause/resume progress events, and persists paused session status under `.claude/workflow-runs/<workflowRunId>/session.json`; live controller cancellation/respawn remains a clean-room approximation gap.

## Suggested implementation priorities

1. Add a hidden-compatible `Workflow` tool facade accepting `{script, name, scriptPath, resumeFromRunId}` while keeping `WorkflowTool` as inspection/control surface.
2. Persist runnable workflow scripts under the Claude session directory shape used by official Claude Code, not only project `.claude/workflow-runs` metadata.
3. Add `scriptPath` edit-and-rerun behavior and expose `workflowRunId` in tool results.
4. Support global `args` as structured data, not only strings.
5. Implement arbitrary JS orchestration primitives (`agent`, `phase`, `pipeline`, logs/progress) instead of only declarative `workflow({ phases })` plans.
6. Add loop/convergence primitives for build/test/fix workflows and iterative review repair.
7. Add first-class reviewer/refuter/voting helpers for independent attempts plus adversarial validation.
8. Align default caps with official limits or document why local recovery defaults are lower.
7. Align spawned agent permission defaults with official `acceptEdits` semantics, or intentionally expose this as a safer local divergence.
8. Add user-level `~/.claude/workflows/` discovery with project workflow precedence.
9. Build an interactive `/workflows` detail view or a closer textual equivalent with phase/agent drilldown and restart/stop/save controls.
10. Mirror official progress event schema names (`workflow_progress`, `workflow_agent`, `workflow_phase`, `workflow_log`) in persisted run output for easier compatibility tooling.
