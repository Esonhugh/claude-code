## Validation Rules

1. **No guessing.** Every feature or fix claim must cite one of: commit, changed file, unit test output, build output, interactive transcript, debug artifact, or pane capture.
2. **Layer labels are mandatory.**
   - `assistant-side`: direct tools in this session, such as `Read`, `Grep`, `Bash`, `InteractiveTerminal.open/write/read`.
   - `binary-side`: behavior inside `built-claude`, such as slash commands, model tool selection, bundled skills, hidden prompt-driven tool use, StopHook behavior, compact behavior, and workflow tool calls.
3. **Interactive-first for binary surfaces.** If behavior exists only inside the Claude binary, do not verify it by source reading alone. Start `built-claude`, type the slash command or prompt, and inspect the terminal/debug output.
4. **Prompt-triggered hidden features must have explicit prompts.** Hidden model-internal skills and tools are validated by prompts that should cause the model to use or avoid the feature.
5. **Positive and negative coverage.** Each feature/fix needs at least one positive path and one negative/regression/control path unless the change is documentation-only.
6. **Evidence hygiene.** Store full terminal captures and debug logs under `/tmp/cc-post-2.1.174-validation/`. Redact tokens, cookies, request bodies, account IDs, and private prompts before summarizing.
7. **No false pass.** If a prompt stalls, a model chooses a different valid path, a permission dialog blocks, or a network/auth dependency is missing, record it as `blocked` or `inconclusive`, not `passed`.
8. **All feature flows must run.** Unit tests are supporting evidence only. The final result must include at least one full binary-side flow for each non-documentation feature family.
